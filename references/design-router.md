# Design Router

The Design Router translates presentation intent into reusable design parameters.

It prevents the workflow from creating one-off themes for every case. A deck should be generated by combining scenario, audience, goal, mood, layout pack, component pack, icon policy, image policy, and design tokens.

## Job

Given the artifacts from Skill 1-3, create a deck-level `design-brief.json` before Skill 4 builds the PPTX.

```text
goal + audience + scenario + mood + content types + brand constraints
-> design mode
-> layout pack
-> component pack
-> color system
-> typography scale
-> icon policy
-> decoration policy
-> image policy
-> density and rhythm
```

The router should not invent slide content. It only decides how the existing slide content should look and what reusable visual tools Skill 4 should use.

## Inputs

Use:

- `00-intake/user-brief.md`
- `01-research/research-brief.json`
- `02-structure/structure-plan.json` or `structure-plan.md`
- `03-production/slide-production-spec.json`
- user-provided brand/style constraints

Relevant signals:

| Signal | Examples | Design impact |
|---|---|---|
| Scenario | business report, product launch, brand event, course training | design mode and layout pack |
| Audience | executives, customers, children, parents, students | typography, density, tone, interaction |
| Goal | inform, persuade, teach, sell, rehearse, celebrate | visual hierarchy and component choice |
| Mood | premium, playful, warm, technical, serious, energetic | color, shape, decoration, image policy |
| Content type | data, process, story, demo, activity, product | component pack |
| Brand constraints | primary color, logo, forbidden styles | color system and asset policy |

## Design Modes

Use these stable design modes. Do not create case-specific modes unless the product deliberately expands the standard.

| Design mode | Use when | Visual character |
|---|---|---|
| `business-report` | work report, business review, strategy update | restrained, evidence-led, chart-safe |
| `executive-briefing` | leadership decision, board update, concise status | sparse, direct, high signal |
| `product-launch` | new product, feature release, demo keynote | large visuals, strong reveal rhythm |
| `brand-event` | public event, campaign, store activation, festival | friendly, branded, image/icon rich |
| `course-training` | tutorial, class, workshop, internal training | clear steps, examples, practice blocks |
| `knowledge-sharing` | public talk, topic sharing, concept explanation | narrative, examples, concept cards |
| `sales-pitch` | proposal, customer presentation, solution selling | pain-solution-proof-action |
| `data-story` | research, market analysis, metrics deep dive | charts, comparisons, source hierarchy |

## Routing Rules

1. Scenario decides `designMode`.
2. Audience adjusts density, language scale, and interaction.
3. Goal decides the dominant component pack.
4. Mood adjusts colors, shapes, decoration, and image policy.
5. Brand constraints override default colors but not readability.
6. Content type decides page-level component preferences.

Example:

```text
brand event + parents/children + store participation + playful/warm + brand blue
-> brand-event
-> event-storytelling layout pack
-> activity-components component pack
-> rounded-line icons
-> medium decoration
-> selective native illustration
```

## Anti-Patterns

- Do not map every deck to the same Swiss/business renderer.
- Do not create a new theme per customer, product, or event.
- Do not treat color changes as a complete design change.
- Do not add decorative images when components would explain better.
- Do not ignore audience mood. A children's event and executive report should not share the same visual grammar.

## Output

Write the result to:

```text
projects/<deck-slug>/04-design/design-brief.json
```

The schema is defined in `design-brief.md`.
