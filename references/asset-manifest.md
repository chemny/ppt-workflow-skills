# Asset Manifest

`asset-manifest.json` records every image, screenshot, chart, generated visual, or external material used by the deck.

## Recommended Location

```text
projects/<deck-slug>/04-design/asset-plan.json
```

## Example

```json
{
  "assets": [
    {
      "id": "hero_product_visual",
      "type": "user-provided-image",
      "status": "available",
      "path": "assets/hero_product_visual.png",
      "purpose": "support the product positioning slide",
      "requiredBySlides": [4],
      "source": "user-provided",
      "license": "user responsibility",
      "required": true,
      "fallback": "use native diagram if missing"
    },
    {
      "id": "workflow_diagram",
      "type": "native-diagram",
      "status": "build-native",
      "purpose": "explain the 4-step process",
      "requiredBySlides": [7],
      "source": "generated by deck builder",
      "required": true,
      "fallback": "text-only process cards"
    }
  ]
}
```

## Status Values

| Status | Meaning |
|---|---|
| `available` | File exists and can be used |
| `pending` | Asset is required but not yet available |
| `generated` | Generated by an image tool |
| `sourced` | Found from a documented source |
| `build-native` | Built as editable PPT shapes/components |
| `placeholder` | Temporary placeholder, must be reviewed before delivery |
| `missing` | Required asset is absent |

## Rules

- Skill 3 should create the first asset plan from `03-production/slide-production-spec.json`.
- Skill 4 should update statuses after build.
- Skill 5 should use the manifest to check missing assets, placeholder leakage, and source risk.
- Generated images must have a prompt, purpose, and fallback.
- User-provided files must be labeled as user-provided; do not invent provenance.
- Optional enhancement assets should set `required: false`; they should warn but not block delivery when a native fallback is already built.
